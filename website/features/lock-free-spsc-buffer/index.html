Lock-Free SPSC Buffer

Chau7 connects reader and parser with one lock-free ring buffer. Zero contention.

What is Lock-Free SPSC Buffer in Chau7?

The Lock-Free SPSC Buffer in Chau7 is a single-producer single-consumer ring buffer that transfers PTY data between the reader thread and the parser thread with zero contention. Chau7 uses atomic cursors instead of mutexes, eliminating kernel synchronization from the hottest data path in the terminal.

The PTY-to-parser pipeline is the path through which every byte of program output flows in Chau7. A mutex on this path would create contention between the PTY reader and the parser, adding latency on every I/O cycle. Chau7's lock-free design avoids this entirely.

How does Chau7 build a lock-free buffer for terminal PTY data?

Chau7's SPSC ring buffer uses two atomic cursors. A write cursor is advanced by the reader thread, and a read cursor is advanced by the parser thread. Each thread only writes its own cursor and reads the other, so there is no contention on any shared memory location.

The buffer in Chau7 is sized to a power of two (typically 256 KB), allowing index wrapping with a simple bitwise AND instead of a modulo operation. Cache-line padding separates the two cursors to prevent false sharing on ARM and x86 architectures, ensuring neither thread's writes invalidate the other's cached cursor value.

What happens if Chau7's ring buffer fills up?

If the parser falls behind the reader in Chau7, the reader detects a full buffer via the cursor positions and applies backpressure by pausing the PTY read. This is the correct behavior for a terminal emulator.

Pausing the PTY read in Chau7 signals the child process to slow down via standard POSIX flow control. This prevents unbounded memory growth and keeps the system stable under burst output. The buffer size of 256 KB provides roughly 3ms of buffering at 80 MB/s throughput, which is large enough to absorb normal scheduling jitter.

Why does Chau7 use SPSC instead of a multi-producer or multi-consumer buffer?

Chau7's PTY data path has exactly one producer (the reader thread) and one consumer (the parser thread). An SPSC buffer exploits this fixed topology to eliminate all atomic read-modify-write operations, using only atomic loads and stores.

Multi-producer or multi-consumer queues require compare-and-swap loops that are strictly more expensive than atomic loads and stores. Since Chau7's PTY data path will always have exactly one reader and one parser, the SPSC design is the optimal choice for this topology.

How does Chau7's lock-free SPSC buffer compare to other terminals?

Most terminal emulators use mutex-protected queues or pipe-based communication between their PTY reader and parser. These approaches introduce kernel transitions and contention on every I/O cycle.

Chau7 uses a lock-free SPSC ring buffer with atomic cursors and cache-line padding, eliminating all contention and kernel transitions from the PTY data path. This design is one reason Chau7 achieves higher throughput than terminals that use traditional synchronization primitives.
