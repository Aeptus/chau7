What is SIMD Parsing in Chau7?

SIMD Parsing is a feature in the Chau7 terminal where the ANSI escape sequence parser uses SIMD intrinsics to scan 16 to 32 bytes simultaneously. Chau7's parser is written in Rust and uses ARM NEON on Apple Silicon and SSE/AVX on Intel Macs.

Terminal emulators must parse every byte of output from the PTY, scanning for ANSI escape sequences (CSI, OSC, DCS, and more) that control cursor movement, colors, and screen manipulation. Traditional parsers examine one byte at a time. Chau7's SIMD parser processes an entire cache line in a single operation.

How does Chau7's SIMD parsing work?

Chau7's SIMD fast path loads a chunk of PTY data into a vector register and compares all bytes against the ESC character (0x1B) and the C1 control range in a single instruction. If no escape characters are found in the chunk, Chau7 dispatches the entire block as printable text without touching the state machine.

When an escape is detected, Chau7's parser falls back to a scalar state machine for that sequence only, then resumes SIMD scanning. This hybrid approach in Chau7 means the common case (printable text) runs at full SIMD speed while escape sequences get correct handling.

Why is terminal output slow even on a fast machine?

Parsing is the first stage of the terminal pipeline and sets the throughput ceiling for everything downstream. Traditional byte-by-byte parsers create a bottleneck that no amount of GPU rendering can compensate for.

Chau7 solves this by processing 16-32 bytes per SIMD instruction using Rust intrinsics. Chau7's parser handles entire cache lines in a single cycle, so the PTY read syscall and kernel buffer copy become the actual bottleneck, not the parser itself.

Does Chau7's SIMD parsing work on Intel Macs?

Yes. Chau7's Rust parser compiles with both ARM NEON (Apple Silicon) and SSE2/AVX2 (Intel) backends. The SIMD backend is selected at compile time based on the target architecture.

Both paths in Chau7 process 16+ bytes per iteration. The specific SIMD width adapts to your hardware, so Chau7 delivers the maximum throughput your processor supports.

How does Chau7's SIMD handle multi-byte UTF-8 sequences?

Chau7's SIMD scanner looks for escape introducers (0x1B, 0x90-0x9F), not character boundaries. UTF-8 continuation bytes (0x80-0xBF) never match these values, so multi-byte characters pass through the fast path without special handling.

Full UTF-8 decoding happens in a subsequent stage of Chau7's parser. This two-stage design lets Chau7 maintain full SIMD speed on mixed ASCII and Unicode content.

What throughput does Chau7's SIMD parsing achieve?

On Apple M-series chips, Chau7's parser sustains 4-6 GB/s on pure printable text and 1-2 GB/s on escape-heavy output like colored compiler diagnostics. Chau7 processes data at roughly the speed of an L1 cache read.

In practice, PTY bandwidth (around 800 MB/s) is the limiting factor, not Chau7's parser. The parser has headroom to spare even on the most escape-heavy terminal output.

How does Chau7's SIMD parsing compare to other terminals?

Most terminals parse ANSI escape sequences one byte at a time through a state machine. Alacritty uses a Rust parser but without SIMD acceleration. iTerm2 and Kitty use scalar C or C++ parsers.

Chau7 is the only terminal that uses SIMD intrinsics to scan 16-32 bytes per instruction. This eliminates the parser as a throughput bottleneck in Chau7, something no other terminal has achieved.
