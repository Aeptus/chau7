## [Unreleased]

### Added
- **MAGI: Multi-Agent Decision Protocol And CLI Foundation**: Added the core MAGI protocol models, first-run persona/config planning, structured council rounds, approved-evidence collection contracts, majority/veto verdict logic, and replay/share artifact rendering for a three-member decision council launched through Chau7 MCP. The new CLI surface is designed to run as `magi`/`MAGI`, ask questions, doctor/configure the setup, replay saved runs, and generate local share artifacts without hosted upload.
- **MAGI CLI Commands**: Packaged `magi` as a SwiftPM executable with `ask`, `doctor`, `config`, `replay`, and `share` flows. The CLI connects to Chau7 MCP for tab orchestration, writes repository-scoped run artifacts when launched inside a repo, falls back to the user's MAGI run store elsewhere, and keeps share output local in v1.
- **MAGI Production Run State And Artifacts**: Added a tested MAGI run state machine for checkpoints, failed/interrupted status, failure categories, denied-evidence counts, and deterministic run IDs. The artifact writer now lives in Chau7Core, writes the complete required bundle, and renders failure metadata into decision markdown, replay JSONL, graph JSON, terminal replay, and local share HTML so partial runs remain inspectable.
- **MAGI CLI Production Hardening**: `magi ask` now writes a failed preflight run artifact when Chau7 is not running or the MCP socket is missing, checkpoints orchestration after each durable stage, categorizes provider/tab/time-out/JSON/evidence/interruption failures, and handles partial artifacts by replaying from `replay.jsonl` or reusing an existing `share.html` when `decision.json` is unavailable.
- **Remote (iOS): In-App Verbose Diagnostics Log With Export And Keystroke Capture**: There was no way for users to capture what the Remote app was doing when input "didn't send" or a tab failed to appear ã only Console.app on a tethered Mac. A new on-device `DiagnosticsLog` records a bounded (8000-entry), disk-persisted ring of structured events across connection lifecycle, input/send attempts (with the block reason), tab-list updates, app lifecycle, and **performance samples** (resident memory via `task_vm_info`, thermal state, uptime, low-power). When enabled, it also records **every keystroke** typed in the app ã text-field character deltas plus the control-key bar (esc/^C/arrows) ã described human-readably (`<return>`, `<esc>`, `<delete N>`). Keystroke capture is privacy-gated: it ships **off** and is enabled only after an explicit **first-run consent prompt** (or by toggling it in Settings), so a fresh install never records typed secrets before the user has seen the disclosure. Settings ã Diagnostics exposes a live, filterable viewer (by level/category/search) and a one-tap **Export** to a plain-text file via the share sheet. Everything stays on-device; nothing is uploaded until the user explicitly exports. New `DiagnosticsLog`/`DiagnosticsLogView` files registered in the iOS target.
- **Remote: Tab List Now Carries Repo, Branch, And AI Provider (`ai_provider`)**: The `TAB_LIST` descriptor gained an optional `ai_provider` field (sourced from each tab's detected AI tool / provider on macOS) alongside the existing project/branch names, so the iOS tab picker can present full context per tab rather than a bare title.
- **MCP: `agent_launch` ã One-Call Multi-Agent Orchestration, Plus Discoverable Launch Tools**: The MCP server exposed only low-level tab primitives (`tab_create` / `tab_exec` / `tab_wait_ready` / `tab_submit_prompt`), so launching agents meant wiring them together by hand ã and a tool search for "launch agent" surfaced *other* MCP servers because none of Chau7's tab tools mentioned agents (a real discoverability miss reported against PR-review workflows). Two fixes: (1) `tab_create` / `tab_exec` / `tab_submit_prompt` descriptions now name AI agents and the launch flow, so keyword search bridges intent ("launch agents") to mechanism. (2) A new `agent_launch` tool fans N agents (Claude Code, Codex, ãÎ) across fresh tabs in one call ã composing `tab_create` ã `tab_wait_ready` ã `tab_exec` (+ best-effort prompt injection once the agent attaches). Params: `directory`, `agent_command` (default `claude`), `prompt`, `count`, `pr_number` (each tab runs `gh pr checkout <pr> && <agent>`), `window_id`, `ready_timeout_ms`; returns the created tab IDs (collect output via `tab_output(source='pty_log')`). Prompt delivery is reported per agent (`sent` / `agent_not_detected` / `skipped`) rather than silently assumed, and `count` is capped by the MCP tab limit. Excluded from `EVAL_MCP_CONTRACT.md` by design ã it's a convenience layer with a best-effort (non-deterministic) injection step, not an eval-critical deterministic surface. New tests cover the PR-checkout command composition; the MCP suites (`MCPSessionTests`, `TerminalControlServiceTests`) stay green at 42 tests, including the `tools/list` schema validation.

### Security
- **Relay: Scoped, Single-Use, Fail-Closed Authentication Across The Worker, Go Agent, And iOS App**: An audit of the Cloudflare relay found auth that failed *open* by default (a forgotten/placeholder `RELAY_SECRET` ran the relay wide open), accepted the token in the URL query string (where it leaks into logs/proxies and could be replayed within a 5-minute window to take over a peer's WebSocket slot), and applied no rate limiting ã while the iOS app sent no token at all. The token scheme is now **v2**: `v2.{ts}.{nonce}.{scope}.{sig}` signed over `v2:{deviceId}:{role}:{scope}:{ts}:{nonce}`, binding each token to one device, role (`mac`/`ios`), and endpoint scope (`connect`/`push`/`pending`), valid 120s, carried **only** in the `Authorization: Bearer` header. The relay verifies in constant time and the Durable Object enforces the `nonce` **single-use**, defeating capture-and-replay/connection-takeover. The Worker now **fails closed** ã absent a configured secret it returns 503 unless `RELAY_ALLOW_UNAUTHENTICATED="true"` is explicitly set for rollout (logged). The shared secret is delivered to iOS through the existing pairing payload (`relay_secret`), so the iOS app (which previously did not authenticate) now mints tokens for its `connect` and `pending` calls; the Go agent moved its WebSocket token from the query string to the header. Relay token mint/verify, base64url, payload validation, APNs-removal, and rate-limit logic are covered by 43 new Node tests; the Go agent gains tests pinning the v2 wire format to the verifier.

### Changed
- **Relay: Reliability & Efficiency Hardening ã Hibernatable WebSockets, Backpressure, Rate Limits, Input Caps, And Correct APNs Token Eviction**: Alongside the auth rework, the relay Durable Object was rebuilt on Cloudflare's **Hibernatable WebSockets API** so an idle macãiOS relay no longer pins the DO in memory or accrues duration billing and survives DO eviction. New protections: per-device, per-route **token-bucket rate limiting** (connect/push/pending); a 64 KB request-body cap with safe JSON parsing (malformed bodies now return 400, not 500); bounded, sanitized pending-state (capped array counts and per-field lengths) so a client can't grow relay storage without limit; a 1 MB relayed-frame cap with **backpressure** (slow receivers are shed instead of buffering unbounded); and a one-shot read-modify-write for dead-registration cleanup that removes a lost-update race. A real bug is fixed: a single oversized/malformed push made APNs return `400`, which the relay treated as "token invalid" and **deleted the victim's push registration** ã a one-request denial of push. Removal is now keyed on the APNs `reason` (only `Unregistered`/`BadDeviceToken`/`DeviceTokenNotForTopic`/`TopicDisallowed`), and the alert payload drops the contradictory `content-available: 1` silent-push flag. The plaintext `RELAY_SECRET` was removed from `wrangler.toml` (it must be a Worker secret), the compatibility date was bumped, and Worker observability was enabled.

- **Rendering: Unfocused-But-Visible Tabs Throttle Their Redraw (new "Background View Refresh Cap"), And The Dead "Active Tab Refresh Cap" Now Actually Works**: A perf report flagged a visible-but-not-focused terminal view (e.g. the selected tab of a non-key window on a second screen) doing ~441 draws + ~287 MiB of grid-sync in a single 30 s window ã the same up-to-120 fps cap as the focused view, for content the user isn't interacting with. Root cause: `minSyncInterval` was a global `1/120` constant that ignored interactivity, and the existing `activePollingRateCap` setting ("Active Tab Refresh Cap") had **zero runtime consumers** ã a dead control whose help even claimed "idle tabs throttle automatically." Both are now wired through one interactivity-aware function: the focused view honors `activePollingRateCap` (`displayNative` ã up to 120 fps, or 60/30), and a visible non-focused view throttles to the new **Background View Refresh Cap** (Settings ã Terminal ã Rendering, default **42 fps**, 1ã120). Crucially the non-focused view stays **event-driven** (PTY-responsive, so switching to it is instant and the deliberate second-screen-streaming fix is preserved) ã only its present/sync rate drops, cutting GPU + grid-sync cost for that view by ~3ã8û. Focusing a throttled view forces one immediate sync so it's never stale. New `TerminalSyncRateTests` pin the rate function (active-cap passthrough, inactive throttle, clamp bounds).
- **Notifications: Defaults Now Fire Only For Agent Approval/Feedback And Completion ã And Bounce The Dock**: Out of the box, notifications also fired for every shell command completion/failure, the AI "idle" heuristic, and mid-run tool failures ã far more than wanted. The shipped default trigger set is now scoped to the two moments that matter: the agent is **waiting on you** (`permission` / `waiting_input` / `attention_required` / `elicitation`) and the agent has **finished working** (`finished` / `failed` / `response_failed`). Shell `command_finished`/`command_failed`, AI `idle`, and `tool_failed` are off by default (still toggleable in Settings ã Notifications). All of those default notifications now **bounce the Dock icon** ã informational (one bounce) for completion, critical (bounces until focused) for approval/feedback. Only the shipped defaults change; any customized trigger/action settings are preserved via the existing override layers. 3127 package tests (pre-existing `FileTailer`/output-queue timing tests flake under heavy parallel load).

### Fixed
- **MAGI Config Wizard Visibility**: The first-run wizard now renders provider and class choices as explicit numbered lists before each prompt, and non-interactive launches explain the detected stdin/stdout TTY state instead of looking like a silent or blank config flow.
- **FileTailer: A Line Written Immediately After `start()` Is No Longer Dropped**: `FileTailer.start()` captured its baseline byte offset *asynchronously* on its private queue, so a write that landed between `start()` returning and the queue running (a monitor that starts then a history/event line lands, or a test that appends right after) could race the offset capture ã `startOnQueue` would read `currentFileSize()` *after* the append, set the offset past the new line, and skip it permanently (kqueue had already fired; the safety poll sees no growth). `start()` now captures the offset and arms the watch synchronously before returning, closing the window. This is what made the `MonitorLifecycleIntegrationTests` (HistoryIdleMonitor) flake under full-suite load and, via the leaked-timer churn it caused, the intermittent `MarkdownRunbookInlineRenderingTests` abort; the full suite is now green across repeated runs.
- **Remote: Multi-Window Macs Now Expose Every Tab To iOS (Not Just The Primary Window's)**: `RemoteControlManager` was wired to a single `overlayModel` (the primary window), so when more than one Chau7 window was open, the iOS tab list, activity feed, interactive prompts, input routing, and snapshots all silently ignored the other windows' tabs. All of these now resolve across **every** registered overlay window via `TerminalControlService.allModels`: the tab list aggregates (de-duplicated by id), input/snapshot/tab-switch look the target up in whichever window owns it, AI-state subscriptions cover all windows, and a new `overlayTabsDidChange` broadcast refreshes the remote list when a non-primary window's tabs or selection change. The remote viewer's selected tab is now tracked independently (`remoteSelectedTabUUID`) of any window's own focus, so iOS can drive a tab living in a background window.
- **Remote (iOS): Terminal Input Now Actually Sends, And Failures Are Visible**: In the multiline (`axis: .vertical`) input field the Return key inserted a newline instead of firing `onSubmit`, so pressing return appeared to do nothing ã the core "input text isn't sent" report. A trailing newline is now treated as a send (when hold-to-send is off), the send button uses an explicit action, every send attempt is logged with its outcome, and a successful send clears any stale "blocked" banner. Blocked sends now report a specific reason (session-not-ready / tab-unavailable / encrypt-failed) to both the error banner and the diagnostics log.
- **Remote (iOS): Header Shows Connection Status Instead Of A Static Title**: The terminal tab's navigation bar showed a fixed "Chau7 Remote" title; it now shows the live connection status (status dot + text) in the title position, with **Pair** (leading) and **Connect/Disconnect** (trailing) actions, and a slim error banner that only appears when there is an error to report.
- **Remote (iOS): Full-Width Tab Picker With Repo And AI Provider**: The tab selector was a narrow capsule menu showing only the tab title. It's now a full-width control that opens a picker sheet listing every remote tab as a rich, full-width row ã tab name, repo name, branch, and AI provider ã with the active tab checked.
- **Rendering: ASCII Glyph Fast Path Removes Per-Cell Allocation From The Metal Render Hot Path**: `updateInstanceBuffer` resolved every cell's glyph through a `Data`-keyed dictionary, allocating a fresh `Data` (and hashing it) per cell per frame ã ~21k heap allocations *per frame* for a full 273û79 grid of ordinary text, the per-frame churn a render sample flagged (alongside 3.1M glyph lookups). Single-byte printable ASCII (the vast majority of cells) now resolves through a flat 512-entry array keyed by `byte<<2|style` with no `Data` and no dictionary hash; multi-byte clusters (emoji, box-drawing, wide CJK, ligatures) keep the existing cache path unchanged. The flat cache resolves lazily via the slow path and clears on atlas reset. 14 renderer tests green.
- **Rendering: Ligature Cache No Longer Thrashes (Real Ligatures Protected From Miss Churn) + Display-Reconfig Re-Clamp Is Debounced**: Two perf-report follow-ups. (1) The Metal ligature cache stored *misses* (every 2ã3 char run that isn't a ligature ã an effectively unbounded keyspace of ordinary text) in the same 4096-entry map as real ligature *hits*, so transient miss churn constantly hit the cap and evicted the small, stable set of genuine ligatures ã re-rasterizing them into new atlas slots (the ~24 cache trims/6 min the report observed). Hits and misses now live in separate caches: real ligatures are kept in their own map (font GSUB û 4 styles ã small, self-bounding) and can never be evicted by miss churn, while misses are FIFO-bounded on their own. (2) `didChangeScreenParametersNotification` fires in bursts during a display reconfiguration (resolution change, monitor plug/unplug, ProMotion switch); re-clamping every overlay window per event is wasteful, so the re-clamp is now coalesced behind a 250 ms debounce. Renderer + memory-policy suites green (16 tests).
- **Detection: A Plain Shell Can No Longer Be Misdetected As An AI Tool From Incidental Output**: A shell tab whose last command was `git push` showed up as "ChatGPT". Root cause: the live output-pattern detector scans the last ~2 KB of terminal output against per-tool fingerprints, ChatGPT's fingerprints included the bare substring `openai.com/v1` (the OpenAI REST endpoint that appears in any project's code/logs/diffs ã the Mockup repo literally contains it), and `AIDetectionState` accepted an output match as *originating* a brand-new identity on a tab with no established AI context. Two fixes: (1) output matching now only **originates** a new identity when a live signal corroborates it ã a child process in the tab's process tree matching the tool ã so incidental text can't flip a plain shell; recognized launches still originate through the high-confidence command path, and output still freely *confirms* an existing identity. (2) Purged the incidental-text fingerprints: the `openai.com/v1` / `openai.com/v1/chat` API-endpoint patterns and the bare `claude.ai` website mention, which were detection liabilities rather than CLI banners. New tests cover the corroboration gate and assert the OpenAI endpoint no longer detects ChatGPT. 3127 package tests.
- **Editor: No Spurious Horizontal Scroll Bar In The Side-Panel Text Editor**: With word wrap on (the default), the editor still showed a horizontal scroll bar ã the scroller was hardcoded on regardless of wrap state, and the line-number gutter never re-tiled the scroll view when its `ruleThickness` changed, letting the document view keep its old width and spill under the wider gutter. The horizontal scroller is now tied to word-wrap state (off while wrapping, on only for no-wrap long-line editing), and the gutter calls `tile()` whenever its width changes. 3127 package tests.
- **Logs: Append-Only AI Event Queues Are Bounded At Launch**: The hook-written event queues (`~/.chau7/claude-events.jsonl`, observed at ~159 MB / 400k lines, and `~/.ai-events.log`) are tailed by live monitors but nothing rotated them, so they grew without bound. These are transient signal queues ã the durable record is `runs.db` ã so at monitor start, before the `FileTailer` attaches and while the writer is quiescent, Chau7 now compacts an oversized file to its most recent ~8 MB (trimmed forward to a line boundary so no partial line survives) via a new `LogFileCompactor`. Safe by construction: the tailer seeks to EOF on start so it never re-reads the file, and compaction runs only at launch when no Chau7-spawned AI session is producing events. New `LogFileCompactorTests` cover the compaction, line-boundary trim, under-threshold no-op, and missing-file cases. 3127 package tests.
- **Telemetry: 30-Day Retention For The Run Database (Configurable) Stops Its Unbounded Growth**: `runs.db` stores full turn content and tool-call I/O for every AI run with no retention ã and the 7-day `PRAGMA incremental_vacuum` was a silent no-op because the database was never created with `auto_vacuum`, so it grew without bound (observed at ~973 MB). At launch (deferred, post-first-paint, off the main thread) Chau7 now deletes runs older than a configurable window ã default **30 days**, set in Settings ã Logs & History (`0` = keep forever) ã cascading to their turns / tool_calls / latency samples via `ON DELETE CASCADE`, then runs a full `VACUUM` to reclaim the freed disk (only when rows were actually removed, so the rewrite is never paid for a no-op pass). New tests exercise the prune (cutoff, cascade, disabled, clamp) against a throwaway database ã never the shared store, since `swift test` does not isolate the home. 3127 package tests (one pre-existing `FileTailer` poll-timeout test flakes under heavy parallel load).
- **Telemetry: "Input Latency" No Longer Counts Shell Command Runtime**: The per-session input-latency metric is meant to measure keystrokeãecho (local PTY/UI responsiveness), but it started its timer on *any* non-empty input ã including the Enter that submits a command ã and stopped at the next PTY output. AI sessions cleared the timer on submission; plain shell tabs did not, so submitting `make` / `git` / a build recorded the command's entire runtime as "input latency", producing misleading multi-second p50/p95 spikes (12 of 14 observed spikes were `app=shell`, some near the 10s cap). The timer now starts only for character/edit input and is dropped on submission for all sessions, so the metric reflects actual echo latency. No behavior change to the terminal itself; submission still processes identically. 3127 package tests (one pre-existing `FileTailer` poll-timeout test flakes under heavy parallel load).
- **Rendering: Full-Screen TUIs No Longer Latch A Stale Width When A Resize Lands During Their Boot**: A relative-cursor TUI renderer (Claude Code's Ink, and the same class of bug for any full-screen TUI) reads the terminal width once at boot and only re-reads it on a `SIGWINCH` `resize` event. Chau7 spawns the child at the size available when the tab starts, but a `layout()` pass routinely resizes the view during the child's ~1ã2s boot (toolbar/tab-bar/window-open animation settling). The kernel's `SIGWINCH` for that resize fires *before* the child installs its handler, so the child misses it, keeps the stale width, and ã because Ink only diff-repaints (no full clear) ã its relative `CSI A`/`CSI G`/`EL` cursor math drifts off-by-N for the rest of the session: assistant output draws over the input line, the `ã₤` prompt duplicates onto two rows, and a finished "computing" spinner never clears. Diagnosed by replaying a real Claude PTY capture through the headless emulator ã clean at the native 135 columns, corrupt at 134 ã proving the emulator and Metal renderer are faithful and the fault is purely a width-model desync. Fixed by re-delivering `SIGWINCH` to the PTY foreground process group a short, coalesced moment after each resize (`chau7_terminal_nudge_winsize` ã `tcgetpgrp` + `killpg`; a plain same-size `TIOCSWINSZ` is silent on Darwin, which only signals on a dimension *change*), so a late-attached handler still re-reads the authoritative size. New emulator regression tests pin full-width-line wrap (`+1` row, not `+2`), `CR CR LF` separator stacking, ambiguous/spinner glyph width 1, the clean native-width replay, and width-mismatch layout drift. 84 chau7_terminal tests (incl. one pre-existing HUP-drain timing test that flakes under heavy local load), warnings-as-errors build clean.
- **Tests: Ungated The Remaining 46 Dead Test Files (+911 Tests), Fixing The Production Bugs They Surfaced**: 46 test files (~1,000 `func test`s) sat behind `#if !SWIFT_PACKAGE`, so SwiftPM never compiled them ã exactly the stability-critical subsystems (Overlay tab lifecycle, Terminal session, SplitPanes, Migration, History, Runtime, MCP, Scripting, Settings) the audit flagged as under-covered. All are now ungated, repaired against the current Sources API, and made hermetic (cross-suite state isolated in setUp/tearDown: UserDefaults, `FeatureSettings.shared`, `KnownRepoIdentityStore`, the on-disk `TabRestoreBundleStore`, the system pasteboard, `MemoryPressureResponder`/`CTOStateFile` overrides). The package suite goes from 2209 ã **3120 tests, all green**. As the notification-test precedent predicted, the ungated tests caught real latent production bugs ã all fixed: `GitDiffTracker.currentFileModDates` mangled relative paths when the repo root was a symlink (`/var` vs `/private/var`), producing entries like `/privateexample.txt`; `TelemetryStore.finalizeRun`'s UPDATE never wrote the `metadata` column, dropping deferred content-extraction markers; `RemoteViewerMode.handleViewerConnection` created duplicate pending entries (and duplicate approval prompts) for a reconnecting viewer; `ScriptingAPI.handleListTabs` was permanently broken (the `tab_list` array response never decoded, always returning empty); `RuntimeIsolation.expandTilde` mishandled a bare `~`; `OverlayTabsModel.reopenClosedTab` silently dropped the tab's repo group; `TerminalSessionModel.restoreAIMetadata` clobbered freshly-restored lifecycle fields to nil on secondary calls; and `handleShellRepoRootReport` marked a session as a git repo at a bogus path for empty/relative reports. A pre-commit guard now rejects any new `#if !SWIFT_PACKAGE` in `Tests/` so dead gates can't return.
- **Memory/Cleanup: Cache Caps, Clipboard Discipline, And Dead Render API Removed**: (1) Clipboard history persisted unbounded payloads into the UserDefaults plist ã a copied multi-MB log went straight into prefs (the exact bloat class the restore-index work evicted); stored items now truncate at 100KB (the live pasteboard keeps the full text), and `paste()` persists its reorder so history order survives relaunch (it silently reverted before). (2) The Claude/Codex session-resolver metadata caches and `ClaudeCodeMonitor.activeSessions` were unbounded for the process lifetime; the resolvers cap at 256 entries and closed monitor sessions evict after a 5-minute grace period (skipped if the session re-opens). (3) Rust: aborted near-max graphics sequences (Sixel/Kitty/iTerm2) kept up to 64MB of `Vec` capacity per buffer ã abort paths now `shrink_to(4096)`; the decoded-image pending store gains a defensive 16-image cap (oldest dropped, logged) in case a consumer stalls. (4) Dead code removed: `OverlayTabsModel.saveTabState` + the instance backup path (zero callers, with instance-level archive fingerprints duplicating the static multi-window accounting for the same directory), and `OptimalMetalView`'s unused frame-pacing API (a CVDisplayLink callback whose `Unmanaged` self could race deinit, a `renderImmediately()` toggling display sync around a draw, a `nextDrawable(timeout:)` ignoring its timeout ã hazards waiting for a first caller). 2209 package tests, 77 chau7_terminal tests green.
- **Concurrency: Small-Race Tail ã Log Sink, Event-Drain Generations, Deallocation Flag, Poll Diagnostics, Usage Snapshot, Reset/Persist Race**: Six formally-racy stragglers: `Log.sink` (a closure reference read on every log call from every thread, assignable at runtime by the debug console) is now lock-guarded; `TerminalEventDrain` replaces its plain `cancelled` Bool (written on main, read on the drain thread ã UB) with a lock-guarded generation counter, so each run captures its generation and any start/stop invalidates older loops without the fragile `stop(); cancelled = false` reset; `RustTerminalView.isBeingDeallocated` (written in deinit, read by drain threads) is lock-guarded; the trace-only poll counters mutated from both drain paths take a diagnostics lock (only when trace is enabled); `UsageMonitor.refreshNow` snapshots `selectedLatencyTimeRange`/`selectedLatencyProvider` before hopping off-main (the same pattern `refreshLatencySection` already used); and `KnownRepoIdentityStore.reset()` cancels the pending debounced persist ã a `record()` just before reset used to fire ãÊ500ms later and re-create the key the user explicitly cleared. 2209 package tests green.
- **Reliability: Persist-Convention Sweep ã A Dozen Silent Failures Now Speak**: The remaining `try?`-swallows-everything stragglers across second-tier persistence and IPC: `ConfigFileWatcher` logged "created config" even when the write failed (read-only repo/sandbox) ã success logs now gate on the write; `FeatureSettings.mcpProfiles`, `EditorConfig.save`, `DangerousCommandGuard` allowlists (save *and* load), and `ProfileAutoSwitcher` rules all route through `Persist.encodeLogged`/`decodeLogged`; SSH import now returns nil for a corrupt file (previously `0`, indistinguishable from "all duplicates") and export logs encode failures; remote approval-request frames and approval-context decodes log instead of vanishing (a security-adjacent flow must never fail silently); MCP telemetry encode failures still degrade to valid empty JSON but are logged (an empty response that's actually an encode bug was invisible); the Claude statusLine *uninstall* sets `lastErrorMessage` when the hook may still be installed; per-repo `injection.json` corruption is logged like the global rules (a typo'd hand-written rule was silently ignored); scrollback reload distinguishes "no cache" from a failed read; the dashboard's companion-plan skeleton write is logged on failure. Also: `clusterString` now bounds-checks against the cluster buffer length (the one of three consumers that didn't), and three `overlayHosts.last!` force-unwraps after `.append` use the appended local instead. 2209 package tests green.
- **Build: Local Release Hygiene ã Real Versions, Pinned Toolchain, Correct Signing Order, No New Dead Test Gates**: Four local-pipeline fixes (per project policy, all CI remains local pre-commit/pre-push/build ã no hosted CI added). (1) `build-app.sh`/`build-dist.sh` derived nothing: a local `build-dist.sh --pkg` release shipped `CFBundleShortVersionString=1.0` unless the operator remembered `CHAU7_VERSION` ã breaking update checks and crash-report triage. The version now derives from the latest `v*` tag (`-dev` suffix for dev builds), and distribution builds *fail loudly* when neither a tag nor `CHAU7_VERSION` is available. (2) A `rust-toolchain.toml` pins rustc 1.96.0 (+rustfmt/clippy) so the dylibs come from the same compiler on every machine instead of floating per install. (3) `chau7_codesign_with_identity` added `--deep` for bundles ã re-signing all the nested Mach-Os that `chau7_codesign_nested_code` had just carefully signed inside-out, overwriting their per-binary identifiers (and Apple deprecates `--deep`); the inside-out flow now stands alone. (4) The pre-commit anti-slop guard rejects new `#if !SWIFT_PACKAGE` gates in `Tests/` ã the pattern that silently killed ~990 test functions can no longer be reintroduced. 2209 package tests green.
- **FFI: ABI Version Handshake + Struct Layout Probes Pin The Swift/Rust Contract**: The `#[repr(C)]` structs crossing the FFI boundary (GridSnapshot read 60û/s, CellData, DebugState, FFIImageData, FFIShellEvent) are hand-mirrored in Swift with ã until now ã no compile- or run-time verification: one field added on the Rust side without updating the scattered Swift mirrors meant silent memory corruption on every frame. ## [0.3.1] - 2026-06-09

### Changed
- **Process-Tree Scan Goes Native (No Subprocess)**: The live AI-tool process-tree snapshot used to shell out to `ps` on a 1.5s timer ã and a second `ps` scan added with argv-based detection had doubled the spawn rate, making the scanner the app's top CPU consumer in a live sample. The snapshot now enumerates the process table in-process via libproc (`proc_listallpids` + `proc_pidinfo`) and reads argv via `KERN_PROCARGS2` only for interpreter processes (`node`/`python`/`ruby`/`npx`), eliminating the recurring `ps` fork/exec entirely. A single `ps -axo pid,ppid,args` scan remains as a fallback if native enumeration ever fails, and a shared `buildSnapshot` keeps both paths producing identical results.
- **Terminal Poll Skips Trace Bookkeeping in Release**: `RustTerminalFFI.pollEvents` runs on the background-drain hot path (up to 60û/s per terminal) and unconditionally did per-poll counter/clock bookkeeping for trace logging even when trace was disabled. That work is now gated behind `Log.isTraceEnabled` (a cached `static let`), so a release build does no per-poll logging work.
- **Background Terminal Drain Backs Off When Idle**: The single 1 Hz timer that drains PTY output for background (deselected/hidden) tabs polled every registered view every second regardless of activity. Dormant views are now polled progressively less often (down to once per ~8 s) via a pure, tested backoff policy that resets to every-tick the instant output resumes, cutting steady background CPU. (The fully event-driven readiness-fd drain ã idle tabs at zero CPU ã needs Rust/FFI changes and is tracked as a follow-up; the active tab already uses the efficient blocking drain.)
- **Memory Pressure Now Reclaims Memory Instead of Only Logging**: OS memory-pressure signals were previously just logged ã no cache shrank in response. A new `MemoryPressureCoordinator` fans the signal out to registered `MemoryReclaimable` caches: `.warning` sheds cheap, regenerable memory while `.critical` releases aggressively, and the bytes reclaimed are recorded on the memory-pressure breadcrumb so the response is observable. Per-tab terminal transcript rings (up to ~10 MB each, fully regenerable backfill data) are the first registrant ã trimmed by half on warning, released entirely on critical. The dead, never-started `MemoryPressureMonitor` was removed.

### Fixed
- **Restore No Longer Drops Tabs After an Unclean Quit**: A regression from the file-bundle restore work made the UserDefaults restore copy write only at termination, so after a force-quit or crash the fallback could be a *stale* index that dropped recently-created tabs ã even whole additional windows ã when the file bundle didn't cover them (this is what lost a non-git tab and its Codex session on relaunch; it was never git-related). UserDefaults now stores a lightweight, **scrollback-stripped index written on every autosave**: tab/session IDs, directory, split layout, AI resume metadata, and panes are preserved (so a fallback restore reconstructs every tab with its `codex resume`), while only the heavy regenerable scrollback/preview payloads are dropped ã those live in the bundle sidecars. This restores the pre-bundle freshness without re-introducing the multi-MB plist writes.
- **Memory Pressure Now Flushes Idle Tabs' Scrollback**: Non-selected tabs were always held in the `.warm` render phase and never reached `.hidden`, so `ScrollbackMemoryManager` never flushed their scrollback to disk ã making non-selected tabs the bulk of a multi-GB footprint in many-tab sessions (the cause of multi-second input stalls observed at 1.2ã2.0 GB with ~20 tabs open). Under sustained OS memory pressure, non-selected tabs are now demoted to `.hidden` (scrollback flushed to disk and the ring shrunk, reloaded on re-selection), via a new `isUnderMemoryPressure` lifecycle input plus a render-lifecycle re-evaluation on each pressure event. The selected tab is never demoted, the PTY keeps draining regardless of phase, and tabs return to `.warm` once pressure clears. This is what lets `MemoryPressureCoordinator` actually reclaim hundreds of MB instead of a few.
- **Live AI-Tool Tab Name Updates Faster Under Load**: The process-tree snapshot that resolves a tab's live AI tool (e.g. detecting a just-launched `claude` and renaming the tab) ran on a `.utility` QoS queue, so under heavy load (verbose logging, many active agents, continuous rendering) the poll and its result were deprioritized and the tab name could lag well past the 1.5 s poll interval. The snapshot queue is now `.userInitiated`; off-tick `refreshNow()` captures are coalesced so a burst of shell events across many tabs can't back up the serial queue; and command-start schedules a short settle-burst of captures (200 ms / 600 ms) so a just-`exec`'d child is detected within a few hundred ms instead of waiting for the next poll tick.
- **Clipboard History Poller No Longer Crashes on `NSPasteboard` Off-Main Access**: The clipboard-history poll timer ran on a background utility queue and read `NSPasteboard.general` (`changeCount` and `string(forType:)`) directly from that thread. `NSPasteboard` is not thread-safe; the off-main reads raced AppKit's pasteboard type cache and intermittently crashed in `objc_msgSend` (`EXC_BAD_ACCESS` / `SIGSEGV` in `-[NSPasteboard _updateTypeCacheIfNeeded]`). The poll now marshals to the main thread before touching the pasteboard; the `changeCount` guard keeps the main-thread cost negligible.
- **Tab Restore Stops Writing Multi-MB Blobs to UserDefaults on Autosave**: Session autosave (every 30s) re-serialized the entire preferences plist on the main thread by writing ~7 MB of tab/scrollback state into UserDefaults ã the source of the recurring large-restore-payload breadcrumb (~176û/day). The file-based, integrity-checked restore bundle is now the primary store and is read first on launch for both the primary window and additional windows; UserDefaults is written only at termination as a one-version downgrade-safety mirror, with on-disk JSON backups kept as the final fallback. The bundle read is all-or-nothing (a corrupt sidecar falls through to the UserDefaults/backup chain), so restore safety is preserved.
- **Proxy High-Water Breadcrumb No Longer Floods**: The `proxy_request_high_water` incident breadcrumb fired on nearly every AI turn ã request bodies grow a few KB per turn as conversation context accumulates, so a strictly-greater high-water gate tripped constantly (~90 `warning` entries/day). It now requires a material step over the prior mark and is recorded at `info` severity: it is diagnostic context correlated against memory-pressure incidents, not a warning condition.

Covered by a new state-machine unit test (`testReadyToFailed_processCrashed`); the existing `testRepeatedApprovalTimeoutsMarkSessionFailed` integration test now actually verifies what it claims.
- **Codex Quits Cleanly Instead of Riding the SIGKILL Path**: AI TUIs (Codex, Claude Code, ãÎ) implement a "Press Ctrl+C again to exit" confirmation, so the single SIGINT the termination ladder sent to the shell process group put Codex in its quit-prompt state but never exited it. The 2 s SIGTERM that followed bypassed Codex's own shutdown handler ã it never tore down the WebSocket session or its `codex-darwin-arm64` PTY child cleanly ã and the chain rode all the way to the 5 s SIGKILL stage on every Codex tab close (logs on 2026-05-08: `close_requested_ms ã 8500`, full SIGINT ã SIGTERM ã SIGKILL ladder, child `codex --model gpt-5.3-codex` left orphaned for the descendant kill). `forceTerminateShellProcessGroupIfNeeded` now schedules a second SIGINT 600 ms after the first one for AI sessions only ã the standard double-Ctrl+C exit pattern ã and surfaces a `second_sigint_ms` field in the `terminationDiagnosticsSummary` so future SIGKILL warnings show whether the second SIGINT actually fired. Plain shells are unchanged (they exit on the first SIGINT) and the SIGTERM/SIGKILL backstops are preserved.
- **Multi-Window Startup Live-Frame Race**: Per-window `OverlayTabsModel` registers its `terminalSessionVisibleFrameReady` observer in `init`, but the SwiftUI terminal view's first paint can fire the notification synchronously *before* the second window's init completes. Idempotent (the coordinator dedups), safe on single-window launches (the observer fires naturally there).
- **Tab cwd Tracks Claude Code Session Directory**: Tabs hosting Claude Code's TUI used to keep their `currentDirectory` stuck at the pre-TUI value when the user `cd`'d inside Claude. - **Wire Metal Coordinator on First switchToView After Init**: `RustMetalDisplayCoordinator.init(terminalView:gridProvider:)` pre-sets `self.terminalView = terminalView`, so the immediate `coordinator.switchToView(focusedRustView, ...)` call from the create branch in `OverlayTabsModel+Refresh.swift` had `newView === oldView`. - **Disable Local-Echo Prediction for TUI Tabs**: Local-echo overlays predicted-input characters at the cursor position and suppresses the matching PTY echo when it arrives, hiding round-trip latency for plain shells. TUI apps (Claude Code, Codex, ãÎ) don't echo typed bytes back as plain ASCII ã they redraw their own input box at a cursor-positioned location, which the prediction layer can't match. Renamed `hostsAIToolForScrollback` ã `hostsAITUI` to reflect its broadened role.
- **Stop Scroll-Storm Flapping Caused by the Throttle's Own Deferred Sync**: Entry into the storm required 3 consecutive high-dirty frames; exit required only 1 low-dirty frame. The throttle's coalesced deferred re-fire (66 ms) lands a sync with whatever's in the triple buffer ã often unchanged, so dirty ã 0. That single low-dirty frame was enough to exit the storm, the next PTY burst re-entered it, and the cycle repeated. 819 enter/exit transitions logged in 2 h on 2026-05-05; effective frame rate stayed ~30 fps instead of the intended 15 fps and the app pressured to ~1.4 GB resident with 7+ s input lag spikes. Exit now requires the same 3-frame run as entry, making the hysteresis symmetric.
- **Codex MCP Config Now Self-Heals on Every Launch**: `upsertCodexMCPSection` previously inserted `command` and `args` only when the `[mcp_servers.chau7]` section was missing entirely; if the section existed but had stale fields (e.g. an `args = ["-c", ...]` from an early-development bridge that took flags, or hand-edits from a debugging session), the writer left those fields alone and Codex would launch the bridge with bogus arguments. The writer now always overwrites both `command` and `args` fields when the chau7 section exists, so any manual or out-of-date setting is corrected on the next Chau7 launch. Multi-line array literals for `args` are not yet recognised.
- **MCP Bridge Survives Chau7 Restarts**: `chau7-mcp-bridge` previously failed-fast if `~/.chau7/mcp.sock` wasn't already listening at startup, and exited the moment the socket dropped. Either condition caused the AI tool's MCP integration to mark chau7 dead until the user restarted the AI session ã common when the user rebuilds and relaunches Chau7 with Claude Code or Codex already running. The bridge now (1) retries the initial connect for up to 30 s, so AI tools that race ahead of Chau7 at boot still attach, and (2) reconnects transparently on socket EOF / error, replaying the saved `initialize` request and any stdin queued during the gap so the AI tool sees an uninterrupted MCP channel across Chau7 restarts.
- **Wrapper Now Updates cwd on Chained `cd X && tui-app` Commands**: The bundled zsh and fish shell-integration wrappers registered `smartoverlay_precmd` only on prompt events (`precmd` / `fish_prompt`). - **Skip Scrollback Flush/Reload for TUI Tabs**: `ScrollbackMemoryManager`'s `flush()` captured the grid via `full_buffer_text` (row text only ã no SGR colors, no cursor positioning, no preserved TUI state). On `reload()` after a hiddenãwarm transition, `replayBuffer` cleared the screen with `ESC[2J ESC[H` and replayed the flattened text. For shell tabs that's correct; for tabs running Claude Code, Codex, Aider, or any other long-running TUI, it destroyed the application's UI invariants ã boxes/spinners/menus that the app had drawn went away, and subsequent in-place updates from the still-running TUI landed on a blank screen, producing visibly corrupted display. The view now flips `hostsAIToolForScrollback` based on the live process-tree signal (`liveAgentName`), and the manager skips flush/reload for those tabs while still resizing the ring (memory tracking unchanged for these phases).
- **Scroll-Storm Threshold Now Catches Near-Full Frames**: The previous classifier required `dirtyCells == frameCells` (every cell dirty) to enter the throttle, which exact-matches a synthetic case but misses the actual workload ã AI TUIs leave a stable status/prompt row, so real streaming sits around 98ã99 % dirty. A Claude session on 2026-05-04 was rendering 21294 / 21567 ã 98.7 % dirty for sustained windows and pressured the app to 1.2 GB resident before being force-quit. Threshold now triggers at ãË 95 % dirty, which captures the real cases without false-firing on normal interactive use (where dirty is typically a handful of cells).
- **Cap Renderer Rate When Whole Viewport Is Dirty**: AI TUIs that stream log-style output (Codex, Claude Code, build watchers) dirty every row of the visible grid every frame, defeating dirty-tracking and uploading a full ~1.6 MiB instance buffer to the GPU per frame. At 30+ fps on a fullscreen viewport that's ~50 MB/s of GPU sync per visible tab ã enough to saturate the Metal command queue and stall the main thread (multi-second input lag observed in 2026-04-30 freeze trace). The Metal coordinator now classifies a "scroll storm" after three consecutive full-grid redraws and throttles `setNeedsSync` to ~15 fps until the dirty rate drops below half the viewport, with a coalesced deferred fire so the final frame of a chunk still renders.
- **Drop Queued Metal Frames for Tabs Switched Away From**: A `metalView.needsDisplay = true` set just before the bound view's `notifyUpdateChanges` flipped to false (the synchronous side-effect of `applyRenderPhase(.warm)` during tab switching) would still trigger one `draw(in:)` at the next CADisplayLink tick, syncing the outgoing tab's grid to the GPU. During rapid tab switching with a chatty AI tab on the previous slot, those leaked frames added tens of MB of sync work per second to an already-saturated Metal command queue. The coordinator now bails early when the bound view is in drain-only mode.

- **Render Lifecycle Focus Hardening**: Visible selected tabs now treat only the key window as input-priority outside startup restore, and selected-tab in-place refreshes reapply the full render phase before repainting.
